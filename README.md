# RAIL

RAIL은 Tauri 기반 로컬 우선 멀티에이전트 워크플로우 데스크톱 앱입니다.  
핵심은 `그래프 캔버스`에서 역할 노드, 데이터/RAG 노드, 게이트/변환 노드를 연결해 DAG를 만들고, 각 노드의 결과를 다음 노드가 받아 종합·검토·문서화하는 흐름입니다.

현재 RAIL은 다음 방향에 맞춰 동작합니다.

- 초보자도 바로 쓸 수 있는 캔버스 중심 UX
- 역할 기반 멀티에이전트 협업
- 첨부 파일/RAG/역할 메모리 기반 컨텍스트 주입
- 내부 품질 가드레일과 워크스페이스별 자기개선 루프
- 로컬 문서 저장, 실행 이력 조회, 웹 연결 보조 실행

---

## 현재 탭 구성

- `그래프`
  캔버스에서 노드를 추가하고 연결해 실행하는 메인 작업면
- `피드`
  실행 결과, 로그, 생성 문서, 실패/저품질 이력을 시간순으로 확인
- `데이터베이스`
  저장된 실행 산출물과 문서를 탐색하고 그래프로 다시 주입
- `개선`
  현재 워크스페이스에서 잘 먹힌 기본 전략과 최근 비교 기록 확인
- `에이전트`
  에이전트 워크벤치와 미션 컨트롤 성격의 실행 공간
- `설정`
  기본 설정, 웹 연결, 메모리 관리
- `인텔리전스`
  Dashboard Intelligence 실행과 토픽별 스냅샷 관리

---

## 그래프 캔버스

그래프 탭은 `질문 직접 입력 -> 역할 노드/데이터 노드 -> 종합/판정 -> 문서화` 식으로 흐름을 만드는 공간입니다.

주요 특징:

- 여러 직접 입력 루트를 둘 수 있고 병렬 실행됩니다.
- 여러 부모 노드의 결과는 다음 노드에 구조화 packet으로 전달됩니다.
- 최종 문서형 노드는 비절단 원칙을 따릅니다.
- 빈 산출은 hard fail 되어 다음 노드로 전달되지 않습니다.
- 노드 추가 시 현재 보이는 캔버스 중앙에 생성됩니다.
- 역할 노드 `내부작업`을 열면 숨겨진 내부 조사/종합/검증 노드를 펼쳐 볼 수 있습니다.

### 노드 종류

- `역할 노드`
  PM, 클라이언트, 시스템, 툴링, 아트 파이프라인, QA, 빌드·릴리즈, 문서화 등 역할 기반 턴 노드
- `데이터 노드`
  RAG/VIA 기반 조사와 자료 수집용 노드
- `변환 노드`
  텍스트 추출, 템플릿 가공, 간단한 구조 재가공
- `게이트 노드`
  PASS/REJECT 같은 분기 판단

### 기본 모델

새로 만드는 기본 turn 노드의 기본 모델은 현재 `GPT-5.3-Codex`입니다.

---

## 역할 노드

역할 노드는 단순 단발성 답변 노드가 아니라, 특정 전문 관점으로 입력을 해석하고 다음 노드에 넘기는 작업 노드입니다.

현재 역할 계열 예시:

- `기획(PM)`
- `클라이언트`
- `시스템`
- `툴링`
- `아트 파이프라인`
- `QA`
- `빌드·릴리즈`
- `문서화`

### 역할 모드

일부 역할은 `창의성` / `논리` 모드를 가질 수 있습니다.

- `창의성`
  발산, 차별화, 대안 생성에 더 적합한 모드
- `논리`
  현실성, 정확성, 리스크, 검증에 더 적합한 모드

중요:

- 창의성은 허황됨을 의미하지 않습니다.
- 비평/종합/최종 문서 노드는 더 보수적으로 동작하도록 설계돼 있습니다.

### 역할 내부작업

역할 노드에는 내부적으로 숨겨진 조사 체인이 붙을 수 있습니다.

기본 흐름:

- 내부 조사 source
- 조사 종합
- 조사 검증
- 최종 역할 노드

이 체인은 평소엔 숨겨져 있고, 노드의 `내부작업` 버튼으로 펼쳐 볼 수 있습니다.

### 자동 리서치

현재 `역할 노드 추가`는 `자동 리서치 포함` 동작입니다.  
즉 역할 노드 하나만 추가해도 실제 실행은 숨겨진 내부 조사 체인까지 함께 실행될 수 있습니다.

이게 의미하는 점:

- 화면에 보이는 노드 수보다 실제 실행 턴 수가 많을 수 있음
- 사용량이 예상보다 커질 수 있음
- 더 강한 grounding이 필요한 역할에는 유용하지만, 가벼운 브레인스토밍엔 과할 수 있음

웹 조사 노드는 자동 내부작업에서 제외됐습니다. 웹 결과가 필요하면 별도 `데이터 노드`나 `WEB 노드`를 직접 추가하는 방식을 권장합니다.

### 추가 요청 저장

우측 역할 워크스페이스는 이제 `실행` 패널이 아니라 `추가 요청 저장` 패널입니다.

동작 방식:

- 선택한 역할 노드에만 추가 지시를 저장
- 저장된 요청은 다음 그래프 실행 때 해당 역할 프롬프트에 함께 반영
- 저장된 요청은 패널에서 삭제 가능

즉 이 패널은 “지금 바로 단독 실행”이 아니라 “다음 역할 실행 때 고려할 추가 지시”를 저장하는 용도입니다.

## RAG / 데이터 노드

RAIL은 그래프 안에서 RAG/VIA 계열 데이터 노드를 함께 쓸 수 있습니다.

예시:

- `source.news`
- `source.community`
- `source.market`
- `source.dev`
- `transform.normalize`
- `transform.verify`
- `transform.rank`
- `agent.codex`
- `export.rag`

주요 특징:

- 첨부 파일은 실제로 노드 실행 시 지식 컨텍스트로 사용됩니다.
- 그래프에 붙인 파일은 on/off 및 제거가 가능합니다.
- RAG 전용 워크스페이스와 역할 내부 조사 체인은 UI상 분리되어 보입니다.
- 웹 결과가 필요할 때는 수동 WEB 노드나 데이터 노드를 추가하는 방식이 더 예측 가능합니다.

---

## 웹 연결

웹 자동화가 필요한 노드는 `WEB` executor를 사용할 수 있습니다.

현재 동작:

- 웹 연결 상태 확인
- 서비스 창 열기
- 프롬프트 복사
- 수동 입력 모달
- 수동 응답 제출

관련 모달은 화면 오버레이 포털로 렌더링됩니다.

- `승인 필요`
- `웹 응답 입력 필요`

WEB 노드에는 노드 카드에서 바로 `수동입력` 버튼이 보입니다.

---

## 메모리

설정 탭의 `메모리 관리`에서 사용자 장기 메모리를 관리할 수 있습니다.

지원 기능:

- 메모리 수동 추가
- 메모리 삭제
- 자동 기억 활성화/비활성화
- 메모리/RAG 활용 활동 확인

메모리는 ChatGPT 웹 메모리처럼 동작하되, 워크스페이스와 그래프 실행 문맥 안에서 보수적으로 사용됩니다.

---

## 개선 탭

`개선` 탭은 내부 평가와 워크스페이스별 자기개선 상태를 보여주는 화면입니다.

사용자에게 보이는 정보:

- `현재 추천 기본값`
- `최근 평가 결과`
- `시험한 다른 방식`
- `관리`

중요:

- 내부 평가/후보 생성/승격 로직은 캔버스에 노드로 노출되지 않습니다.
- 이 탭은 모니터링과 동결/재개/초기화용입니다.
- 이미 저장한 사용자 그래프를 자동으로 다시 쓰진 않습니다.

---

## 품질 가드레일

최근 기준으로 RAIL에는 다음 가드레일이 들어 있습니다.

- 문서형/최종형 노드의 비절단 원칙
- 중략/생략/`...` 표기 품질 실패 처리
- 빈 산출 hard fail
- 구조화된 다관점 입력 packet
- 최종 종합 노드의 grounding/근거/한계 체크
- 워크스페이스 적응형 기본값 학습

즉 목표는 “더 길고 그럴듯한 답변”이 아니라 “역할별 관점을 살리되, 근거 없는 산출은 탈락시키는 것”입니다.

## 1인 인디게임 아이디어용 추천 구조

가장 추천하는 기본형은 아래입니다.

```text
질문 직접 입력
-> 기획(PM, 창의성)
-> 클라이언트 / 시스템 / QA / 아트 병렬
-> 기획(PM, 논리)
-> 문서화
```

이 구조의 의미:

- `기획(PM, 창의성)`에서 초안 생성
- `클라이언트/시스템/QA/아트`가 같은 초안을 각 전문 관점으로 병렬 심사
- `기획(PM, 논리)`가 상충하는 평가를 종합
- `문서화`가 최종 제안서와 프로토타입 계획 정리
