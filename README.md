# RAIL

RAIL은 Tauri 기반 로컬 우선 워크플로우 데스크톱 앱입니다.
현재 버전은 `그래프`, `데이터베이스`, `설정` 3개 탭을 중심으로 동작하며, 그래프 안에서 `DAG`/`RAG` 뷰 전환, 역할 노드 협업, 유니티 자동화 프리셋을 함께 다룹니다.

---

## 현재 탭 구성

- `그래프`: 캔버스 기반 워크플로우 편집 및 실행 (`DAG`, `RAG` 뷰 전환)
- `데이터베이스`: 실행 산출물, 역할 지식, 조사 결과 문서/JSON 조회 및 관리
- `설정`: 런타임/환경 설정

---

## 그래프 모드

### DAG 모드
- 일반 그래프 노드(턴/변환/게이트 등) 구성 및 실행
- 노드 단위 실행 상태와 로그 확인
- 역할 노드는 기본적으로 `전문가 역할 노드 + 내부 조사/종합/검증` 구조를 사용
- 역할 노드끼리 직접 연결해 협업 그래프를 구성할 수 있음
- 역할 노드를 선택하면 우측에 전용 터미널 island가 나타나고, 현재 노드 관련 로그/추가 지시를 확인할 수 있음

### 역할 노드

현재 역할 노드는 `기획(PM)`, `클라이언트`, `시스템`, `툴링`, `아트 파이프라인`, `QA`, `빌드·릴리즈`, `문서화` 관점으로 답변하는 작업 노드입니다.

핵심 동작:
- 메인 DAG에는 기본적으로 `역할 노드`만 보입니다.
- 역할 노드 내부의 조사 체인과 검증 체인은 기본 캔버스에서 숨겨집니다.
- `내부작업 보기`를 누르면 해당 역할 노드의 내부 조사/종합/검증 흐름을 펼쳐서 볼 수 있습니다.
- `추가 시각`을 누르면 같은 역할의 다른 관점 인스턴스를 추가합니다.
- `재검토`를 누르면 기존 역할 결과를 다시 비판적으로 검토하는 패스를 추가합니다.

### 역할 노드 메모리

역할 노드는 단순 일회성 노드가 아니라, 실행이 끝난 뒤 검증 통과한 결과를 역할 지식으로 누적합니다.

- `역할 공통 메모리`: 같은 역할이 공유하는 누적 지식
- `관점 인스턴스 메모리`: 추가 시각/재검토처럼 분기된 관점이 따로 누적하는 판단

이 구조 덕분에:
- `기획 -> 클라이언트 -> 기획`처럼 같은 역할이 다시 등장해도
- 완전 새 노드가 아니라, 이전에 검증된 역할 맥락을 이어받은 다음 패스로 동작합니다.

### 역할 내부 조사

역할 노드를 `자동 리서치 포함`으로 추가하면, 내부적으로 역할 전용 조사 체인이 함께 생성됩니다.

역할 내부 조사 체인은 다음을 조합합니다.
- 레포/프로젝트 맥락 조사
- 외부 문서/실전 사례 조사
- 조사 종합
- 조사 검증

이 내부 조사 체인은 역할별로 다르게 설계되며, 필요 시 `via_flow`를 통해 기존 소스 수집 경로를 재사용합니다.

### RAG 모드
- VIA 계열 파이프라인을 RAIL 내부에 내장한 실행 모드
- 템플릿 기반 파이프라인 생성
  - `주식/마켓`, `커뮤니티`, `뉴스`, `SNS`
- 주요 수집/처리 노드
  - `trigger.manual`
  - `source.*` (`news`, `sns`, `community`, `market`, `dev` 등)
  - `transform.normalize`, `transform.verify`, `transform.rank`
  - `agent.codex`
  - `export.rag`

중요:
- `RAG` 뷰에는 `RAG 전용 노드`만 보입니다.
- 역할 노드 내부에서 자동 생성된 조사용 `source.community`, `source.dev` 노드는 `RAG` 뷰에 섞이지 않습니다.
- 즉 `DAG 역할 협업 그래프`와 `RAG 워크스페이스`는 UI상으로 분리되어 있습니다.

## RAG 동적 크롤링 옵션

RAG에서 `source.*` 노드를 선택하면 우측 워크스페이스에 동적 크롤링 옵션이 표시됩니다.

입력 필드:
- `키워드 (쉼표)`
- `국가 코드 (쉼표, 예: KR,US,JP,CN)`
- `사이트/URL (쉼표 또는 줄바꿈)`
- `최대 수집 건수 (숫자)`

### 동적 크롤링 프리셋 (4종)
- `마켓 핫토픽`
- `커뮤니티 핫토픽`
- `뉴스 헤드라인`
- `SNS 트렌드`

권장 사용 순서:
1. RAG 템플릿 적용
2. 소스 노드 선택
3. 동적 프리셋 적용 또는 직접 입력
4. 실행

주의:
- 템플릿을 다시 적용하면 노드가 재구성되면서 커스텀 옵션이 초기화될 수 있습니다.
- 국가 코드는 지원 목록 외 값이 무시될 수 있습니다.
- 최대 수집 건수는 내부 제한(최대 120)이 적용됩니다.

## 런타임 구조 (내장형 VIA)

외부 `/via` 저장소 없이 RAIL 내부 런타임으로 동작합니다.

- Python 런타임: `scripts/via_runtime/`
- Tauri 브리지 명령:
  - `via_health`
  - `via_run_flow`
  - `via_get_run`
  - `via_list_artifacts`
- 앱 최초 실행 시 워크스페이스 기준 부트스트랩:
  - `/.rail/.venv_via` 생성 및 의존성 준비

데이터 경로(워크스페이스 기준):
- 런타임 데이터: `/.rail/via/`
- RAG 산출물: `/.rail/via-docs/rag/{flow-name}/{run-id}/`

추가로 역할 내부 조사 체인에서 `via_flow`를 사용할 경우:
- 기존 `scripts/via_runtime/server.py`의 소스 어댑터 체인을 재사용합니다.
- 이 경로에는 `scrapling` / `pinchtab` 기반 수집 어댑터가 포함되어 있습니다.
- 따라서 `DAG 역할 조사`도 내부적으로 같은 소스 수집 계열을 활용할 수 있습니다.

## 유니티 자동화

그래프 우측 `유니티 자동화` island에서 다음 프리셋을 사용할 수 있습니다.

- `유니티 CI 닥터`
- `유니티 테스트스미스`
- `빌드/용량 감시`
- `로컬라이제이션 QA`
- `Addressables / 에셋 다이어트`

현재 상태:
- 기본은 보호된 진단/미리보기 흐름입니다.
- 프리셋 선택 시 보호 확인과 샌드박스 준비를 자동으로 진행합니다.
- 유니티 실행 파일 경로를 입력하면, preset별 `빌드`, `EditMode`, `PlayMode`, `Addr 빌드` 같은 명령 미리보기를 확인할 수 있습니다.
- 기본 목적은 원본 프로젝트를 직접 건드리지 않고, 안전하게 진단/자동화 흐름을 구성하는 것입니다.

## 빠른 시작

```bash
npm install
npm run tauri:dev
```

웹 UI만 실행:

```bash
npm run dev
```

빌드:

```bash
npm run build
```

전체 체크:

```bash
npm run check
```

## 프로젝트 구조

```txt
src/
  app/                # 앱 조립, 상태, 런타임 핸들러
  pages/              # 대시보드/그래프/데이터베이스/설정 화면
  features/           # 도메인 로직 (workflow, studio, presets 등)
  styles/             # 페이지/레이아웃 스타일
scripts/via_runtime/  # 내장 VIA Python 런타임
src-tauri/            # Rust bridge 및 시스템 명령
```
